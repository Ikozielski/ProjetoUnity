# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

A Unity project (early prototype stage) for a 2D Pong-style game. Scripts and asset names are in Portuguese (`Raquete` = paddle/racket, `bola` = ball).

- Unity Editor version: `6000.5.2f1` (Unity 6), see `ProjectSettings/ProjectVersion.txt`.
- Render pipeline: Universal Render Pipeline (URP) 2D Renderer (`Assets/Settings/Renderer2D.asset`, `UniversalRP.asset`).
- Input: new Unity Input System (`com.unity.inputsystem`), configured via `Assets/Settings/InputSystem_Actions.inputactions`.
- Only one scene exists: `Assets/Scenes/SampleScene.unity`, containing a Main Camera, a Global Light 2D, and a `Raquete_0` GameObject.
- Only one script exists: `Assets/Scripts/RaqueteController.cs`, currently an empty MonoBehaviour stub (no gameplay logic implemented yet).

## Working with this repo

This is a Unity project, not a typical CLI-buildable project — there are no npm/dotnet/make build or test scripts in the repo. Building, running, and testing happens through the Unity Editor (Play mode) or Unity's command-line batch mode.

- C# scripts live under `Assets/Scripts/` and are plain MonoBehaviours edited as text files. Every asset (script, scene, texture) has a paired `.meta` file with Unity-generated GUIDs — never delete or hand-edit `.meta` files; let Unity manage them, and keep them alongside their asset when moving/renaming files (or Unity will lose references).
- Scene files (`.unity`) and asset files (`.asset`) are YAML. Prefer making structural/GameObject changes through the Unity Editor rather than hand-editing these YAML files directly, since they contain fragile GUID references and serialized layout data.
- `Assembly-CSharp.csproj` and `Projeto Unity.slnx` are Unity-autogenerated (regenerated on editor open) — do not hand-edit them; they're already gitignored for the `.csproj`/`.sln`/`.slnx` extensions but the top-level `.slnx` in this repo predates that ignore rule.
- Package dependencies are managed via `Packages/manifest.json` (Unity Package Manager), not a `package.json`.
- `Library/`, `Temp/`, `Logs/`, `UserSettings/`, `obj/` are Unity-generated caches/derived data and are gitignored — never treat their contents as source of truth or commit to them.
